> **DISCLAIMER.** This document is an internal self-assessment produced for improvement planning. It is not an independent third-party audit, a penetration test, or a certification of any kind. ## Executive Summary

Lionade's security posture is strong on the controls that are expressed in code and weak on the disciplines that live on paper. PROTECT is the standout function: identity and access control are genuinely good for a solo-founder pre-scale SaaS, with a single server-side auth helper covering roughly 169 of 221 API routes, a role-tiered admin gate with audit logging, and a layered database defense (BEFORE UPDATE column guards, SECURITY DEFINER lockdown, and a GRANT revoke on the answer bank) that closes the browser-client privilege-escalation class. Data security is similarly solid, with a server-authoritative reconciled dual ledger, an AES-256-GCM credential vault whose key lives only in the environment, SSE-KMS on S3, and no hardcoded secrets. DETECT and RESPOND have surprisingly mature tooling for a one-person team: an L7 SOC with threat signatures, a bounded telemetry pipeline, a feature kill-switch with auto and manual incidents, an IP denylist, and a genuinely good RFC 9116 vulnerability disclosure policy.

The biggest gaps are governance and process rather than missing controls. There is no documented risk management strategy, risk appetite, threat model, or risk register anywhere in the docs; no named security-accountability role distinct from the founder; no recurring oversight cadence; and effectively no supply-chain risk management (no Dependabot or Renovate, and npm audit currently reports 18 vulnerabilities including 10 high severity). Recovery is the weakest function: core data recovery depends entirely on the unverified Supabase plan tier with no documented restore test, the Terraform remote state is still commented out and lives on a single laptop, and there is no breach-notification path for a user base that includes minors. Because Lionade runs a real-money economy serving minors and is heading toward a V2 cash-payout launch, the highest-priority work is to close the breach-notification and tested-restore gaps and to stand up a basic risk and supply-chain management discipline before that launch raises the stakes.

## PROTECT (PR)

Lionade's PROTECT posture is unusually strong for a solo-dev pre-scale SaaS, with real, code-backed controls rather than aspirational ones. Identity and access control are the standout: a single requireAuth helper covers ~169 of 221 API routes (the rest are legitimately public, cron-secret, Stripe-signature, or internal-secret gated), a role-tiered admin gate (requireRole) protects all 29 admin routes with audit logging and email masking, and a layered DB defense (BEFORE UPDATE column guards in migrations 078/080, SECURITY DEFINER lockdown in 076, an answer-bank GRANT revoke in 081) closes the browser-Supabase-client privilege-escalation class. Data security is genuinely good (server-authoritative economy with a reconciled dual ledger, AES-256-GCM credential vault with the key held only in env, SSE-KMS on S3, no hardcoded secrets, no NEXT_PUBLIC leaks). The two real residual gaps are infrastructure resilience (single-region us-east-1, in-memory rate limiter with no Redis, remote Terraform state still commented-out/bootstrap-pending) and the near-total absence of formal awareness/training and a couple of legacy USING(true) RLS write policies on arena tables.

## Remediation roadmap

The roadmap below sequences the work by urgency and ties each item to Lionade's in-flight security-projects track where relevant. Items are grouped into three waves: the first wave must complete before the V2 cash-payout launch, the second wave is high-value hardening, and the third wave closes the governance loop.

### Wave 1: Before V2 cash-payout launch (P0 plus the highest-leverage P1s)

1. **Tested data recovery and remote state (RC.RP, PR.IR).** Confirm and document the Supabase plan tier and its PITR/backup window, then run and document one real restore to a scratch project with a recorded RTO and RPO. Complete the Terraform backend bootstrap (uncomment terraform/backend.tf, run terraform init -migrate-state) so infrastructure state moves off the laptop, and add point_in_time_recovery to the DynamoDB lock table. Schedule a periodic logical export of the coin_transactions Fang ledger to the versioned S3 bucket. This is the cloud misconfiguration scan (CSPM) of the AWS footprint made concrete: the same pass should verify S3 SSE, public-access blocks, OIDC role scoping, and the now-active remote state.
2. **Breach-notification path (RS.CO, RC.CO).** Draft a breach-communication runbook covering who is notified (users, guardians of minors, the EU DPA within 72 hours, relevant US state attorneys general), the trigger criteria, and two pre-approved notice templates (one in-app and email via the existing Resend pipeline, one regulator filing). Pre-draft a minor-data breach notice so it is not authored under pressure. This complements the already-shipped vulnerability disclosure policy: the VDP (DONE: app/security + SECURITY.md + security.txt) governs inbound researcher reports, and this runbook governs outbound breach communication. Finalize the lawyer-reviewed VDP the policy already promises in the same legal pass.
3. **Incident-response playbook plus tabletop (RS.MA, RS.AN, RS.MI, PR.AT).** Write a one-page IR plan with a severity taxonomy (SEV1 data/economy breach, SEV2 outage, SEV3 minor), a declare-contain-eradicate-recover-review flow, the single-operator role map plus a named backup contact, and target acknowledgement and containment times. Document the deploy-rollback and Supabase PITR steps and add an economy-freeze procedure (a payouts kill-switch key plus a scripted bulk-reversal against the dual ledger). Then run one tabletop exercise against a simulated Fang-minting incident to validate the plan end to end. Point the plan at the existing /admin/features kill-switch, IP denylist, and incidents table for tooling.

### Wave 2: High-value hardening (remaining P1s)

4. **Supply-chain and dependency management (GV.SC, ID.AM, PR.PS, DE.CM).** Enable Dependabot or Renovate, add a CI npm audit (or osv-scanner) gate, and remediate the 10 current high-severity findings now (notably next, ws, and picomatch). Maintain a sub-processor list with DPA status and a process for reviewing vendor advisories. Add a lightweight external error monitor (Sentry free tier) so unhandled exceptions reach a human independent of the bespoke telemetry pipeline.
5. **Least-privilege IAM review (PR.AA, GV.RR).** Run a least-privilege pass over both the application role model and the cloud IAM. On the data layer, close the two legacy permissive arena RLS write policies (arena_matches_update and arena_answers_update) by binding auth.uid() to a participant column or revoking the authenticated write grant. On the cloud layer, review the OIDC-assumed AWS role for any over-broad actions and confirm the role assume condition is scoped to the exact project and environment. Name the security-accountable owner and a deputy/escalation contact as part of this pass.
6. **Detection latency and economy-drift monitoring (DE.CM, DE.AE).** Confirm INTERNAL_TELEMETRY_SECRET and the security_monitoring migration are live in production and document the on/off state. Move the security-alerts and feature-health crons off daily scheduling to a near-real-time cadence. Add a scheduled economy-integrity detector that re-runs the migration-075 dual-ledger drift check and alerts on any nonzero drift or improbable Fang grant velocity. Wire the documented Upstash Redis backend so rate limiting and offender detection are accurate across edge instances, and add a secondary alert channel for high-severity events.
7. **LLM and prompt-injection red-team of the AI features.** Conduct a focused red-team of the AI surfaces (Ninny and any server-side LLM calls) for prompt injection, jailbreaks, data exfiltration through model outputs, and cost-amplification abuse. Tie the findings into the new risk register and ensure the per-route rate limiting on AI endpoints holds once limits are global (see item 6). Treat any discovered injection paths as economy and minors-data risks given that AI outputs can drive user-facing content.

### Wave 3: Close the governance loop (P1 and P2 strategy)

8. **Risk register and threat model (GV.RM, ID.RA).** Stand up a single RISK_REGISTER.md with the top 8 to 10 risks, each scored by likelihood and impact with an owner and treatment decision, seeded from the prior audit Bucket D, the tech-debt list, and the economy-integrity items. ## Methodology and limitations

This assessment was produced by mapping the Lionade codebase, infrastructure-as-code, and documentation against the six functions and their categories in NIST CSF 2.0. Evidence was gathered by reading source files, configuration, and migrations and by running targeted searches (for example grep for risk-register and threat-model terms, RLS enablement counts, secret patterns, and auth-helper coverage across API routes). Each category received a rating of Strong, Partial, or Gap based on whether the relevant control is implemented and effective, present but incomplete or unverified, or largely absent. Priorities (P0 through P3) reflect urgency relative to the planned V2 cash-payout launch and the product's real-money, minors-present risk profile.

The assessment has real limits. It is a point-in-time internal self-assessment, not an independent audit, penetration test, or certification, and it reflects one person's reading of the artifacts. Several controls could only be partially verified from the repository: Supabase backup and PITR guarantees, whether feature-gated pipelines (telemetry, CloudWatch heartbeats) are actually enabled in production, and the live state of the privacy, terms, and disclosure documents, which self-identify as working drafts pending legal review. File paths and line references are preserved from the underlying per-function evidence and were accurate as of the assessment date; subsequent commits may move them. Ratings were assigned conservatively and were not inflated; where evidence was strong but a residual gap existed, the rating reflects the gap. This document is intended as an input to an improvement loop, and the roadmap above should be tracked to closure and the assessment re-run on a recurring cadence.
