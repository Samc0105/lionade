- fix(web): **Hydration mismatch on persisted SWR provider (local commit, NOT pushed).** Blocking-bug fix on commits `3111969`, `089aa9c`, `5bd6445`, `f185fba`. After Phase B's localStorage-persisted SWR cache landed, both `/dashboard` and `/quiz?subject=Science` started throwing `Did not expect server HTML to contain a <div> in <div>` on every refresh for signed-in users with a populated cache. Root cause: `lib/swr-config.ts:getProvider()` constructed the persisted provider on the very first render in BOTH environments. Server SSR has no localStorage so the cache initialized empty and the tree rendered the loading skeleton. The client's first render (the hydration commit) ran the same factory but the synchronous localStorage read populated the cache before useSWR hooks read it, so the client tried to paint the data DOM. The no-persist config mirrors every revalidation/retry setting so the only observable difference across the swap is whether the provider hydrates from disk. Trade-off: SWR cache is re-mounted across the swap so hooks may fire one extra revalidation immediately after hydration. Acceptable to keep server/client DOM aligned. **Why Option A and not Option B (deferred hydration in the shared factory).** The shared `createPersistedSwrProvider` in `packages/lionade-core/src/cache/storage.ts` already does hydration asynchronously (via an internal async IIFE that resolves `readyPromise`), but the localStorage read still resolves on a microtask which races with React's hydration commit. Solving this at the factory layer would require an opt-in `skipInitialHydration` flag plus a web-side `.hydrate()` call from `useEffect`, which touches the shared package and adds API surface. Option A keeps the entire fix inside web with zero change to the shared factory or to iOS, and the cost is one extra revalidation wave right after first paint. **Hard constraints respected.** No edits to `lib/auth.tsx`, `lib/supabase.ts`, or `middleware.ts`.
